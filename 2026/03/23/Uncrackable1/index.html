题目解析

绕过root检测

模拟器上面安装，发现root检测，点ok之后会直接退出。在真机的su是内核层级的，不会被检测。

用jadx打开，直接可以看到有root和debug检测
可以选择用frida直接注入

先进入c类获得包名

  • Java.perform — 告诉 Frida 进入 java 运行时环境，在里面执行操作。
  • Java.use("")是在运行是获取这个java类的引用，在后面可以对他里面的方法进行替换或者调用，等号左边一般直接用这个类名来接住
  • c.a.implementation = function() { return false; }：c.a是指c类的a方法,implementation = function() {}是指用花括号里面的代码来替换a里面原本的代码，return false就是直接让函数返回false，在这里表示没有检测到root
    同样的方法把abc三个方法都直接返回false，就能让程序以为没有root，就可以成功绕过。

可以看出没有弹窗，并且cmd里面也有我们的日志

获得密码

根据文本的提示，a.a显然是判断的函数。前面的代码是获取输入框里面用户输入的文本，并传到a.a()里面；后面就是制作成功或失败的弹窗并显示。

进入a.a()查看，大致逻辑是把我们输入的文本当初str传进来，然后通过sg.vantagepoint.a.a.a（这里不是现在这个a.a，而是另外一个包下的类的方法）把传进去两个奇奇怪怪的东西转化成bArrA，最后对比str和bArrA。显然bArrA就是我们要的密码。

1.在生成密码的时候读取

  • this.a(key, ciphertext)表示调用一遍原来的方法
  • 定义ret接住产生的密码
  • 生成日志打印出来
  • 最后要记得把原来的密码返回，不然程序后续有问题
    在这里其实整个程序执行的过程没有任何改变，我们只是在过程中泄露了密码，这就是this的作用。

然后直接点击验证就可以获得密码

在比较的时候读取

在原来的代码中str.equals(new String(bArrA))，是str调用了equal，所以this指的是调用equals的字符。
为什么要加两个检测呢？因为equals是java自带的库函数，整个程序会有很多地方调用这个方法，我们用implement替换之后任何一个地方调用的都是替换后的函数，所以要加一点限制帮助我们更精确地泄露密码；如果不加两个检测，每次调用都会输出一个日志，满屏日志肯定难以发现密码。这里我选择让用户输入的字符长度为1时输出，所以输入的时候只能输入一个字符。

这次输入一个a，再点验证就可以获得密码

直接生成密码

这里需要再解析一下生成密码的过程
先用同一个类下的b方法处理8d127684cbc37c17616d806cf50473cc,然后用Base64解码5UJiFctbmgbDoLXmpL12mkno8HT4Lv8dlat8FxR2GOc，再把这两个玩意传入sg.vantagepoint.a.a.a这个方法，产生密码。所以直接获取要用的类，按照原来的逻辑直接产生密码即可。

Root检测分析

a() — PATH环境变量中查找su

  • 获取系统的 PATH 环境变量，按 : 分割成多个目录
  • 逐个目录检查里面有没有 su 文件
  • 传统root方案会在 /system/bin/ 或 /system/xbin/ 放一个su二进制文件，这个方法就是针对这种情况
  • KernelSU系列方案不在文件系统放su文件，所以检测不到

b() — 检查系统签名

  • Build.TAGS 表示系统镜像的签名类型
  • 正式发布的系统用 release-keys 签名，开发/测试版本用 test-keys

c() — 检查已知root文件路径

  • 硬编码了一堆老牌root工具的特征路径（SuperSU、Superuser等）
  • 逐个检查这些文件是否存在
